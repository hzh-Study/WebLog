## 一、核心目标与价值定位

### 1.1 业务目标

| 目标维度 | 具体目标 | 量化方向 |
|---------|---------|---------|
| **用户留存** | 提升用户回访率与阅读深度 | 次日留存率提升 15%，人均阅读篇数提升 20% |
| **内容消费** | 增加优质内容的曝光与消费时长 | 文章平均阅读时长提升 25%，长尾内容曝光率提升 30% |
| **平台活跃** | 提高用户互动行为频次 | 点赞/收藏/评论率提升 20%，人均会话时长提升 18% |
| **商业变现** | 为后续广告/付费内容推荐奠定基础 | 推荐位点击转化率达基准线 3 倍 |

### 1.2 利益相关方价值

| 利益相关方 | 核心价值 | 具体收益 |
|-----------|---------|---------|
| **读者用户** | 发现感兴趣的内容，减少信息过载 | 个性化首页、精准关联推荐、减少无效浏览时间 |
| **作者/创作者** | 作品获得更多目标读者曝光 | 文章推荐量提升、粉丝增长加速、互动反馈增加 |
| **平台运营** | 提升整体内容消费效率与用户粘性 | 用户活跃度提升、内容生态健康度改善、运营成本降低 |
| **业务方** | 数据驱动的增长引擎 | 可量化的推荐效果、可配置的运营策略、未来商业化的基础设施 |

### 1.3 与产品战略的对齐

```
WeBlog 产品战略: "打造智能化的个人技术博客平台"
        │
        ├── 内容生产端 ──→ AI 写作辅助（已有 AiWriterPanel）
        │                 ──→ AI SEO 优化（已有 AiSeoWorkbenchPanel）
        │
        ├── 内容消费端 ──→ ★ 智能推荐系统（本次规划）
        │                 ──→ AI 阅读助手（已有 ArticleAiReader）

        └── 平台运营端 ──→ 数据分析仪表盘（已有 Dashboard）

```

### 2.3 模块二：内容匹配算法模块

#### 2.3.1 算法分层架构

```
Layer 1: 召回层（Recall）── 从全量内容中筛选候选集（万 → 千）
    ├── 协同过滤召回（UserCF / ItemCF）
    ├── 内容召回（标签匹配 / 类别匹配）
    ├── 热门召回（时间衰减热度排序）
    └── 实时行为召回（最近浏览/搜索相关）

Layer 2: 排序层（Ranking）── 对候选集进行精细排序（千 → 百）
    ├── 特征工程（用户特征 + 内容特征 + 上下文特征）
    ├── 粗排模型（逻辑回归 / GBDT）
    └── 精排模型（深度学习 / 多目标优化）

Layer 3: 重排层（Re-ranking）── 业务规则与多样性调整（百 → 十）
    ├── 多样性打散（类别/作者去重）
    ├── 时效性加权（新文章提权）
    ├── 业务规则干预（运营置顶/屏蔽）
    └── 冷启动兜底（新品/新用户特别策略）
```

#### 2.3.2 推荐场景与策略

| 推荐场景 | 触发时机 | 推荐策略 | 展示数量 |
|---------|---------|---------|---------|
| **首页推荐流** | 用户访问首页 | 个性化混合推荐（协同过滤 + 内容推荐） | 10-20 条，支持下拉加载 |
| **文章详情关联** | 文章阅读中/结束后 | 基于内容的相似文章推荐 | 3-6 条 |
| **搜索建议** | 用户输入搜索词 | 热门搜索 + 个性化搜索补全 | 5-10 条 |
| **用户主页推荐** | 访问用户主页 | 该作者热门文章 + 相似作者推荐 | 5-10 条 |
| **分类/标签页推荐** | 浏览分类或标签 | 该分类下热门 + 个性化排序 | 列表模式 |
| **侧边栏推荐** | 全局展示 | 热门文章 / 编辑精选 | 5-8 条 |
| **推送通知** | 定时/事件触发 | 关注作者更新 / 兴趣话题新文 | 1-3 条/次 |

#### 2.3.3 冷启动策略

| 冷启动类型 | 策略 | 数据依赖 |
|-----------|-----|---------|
| **新用户冷启动** | 热门推荐 + 兴趣标签引导选择 + 探索-利用平衡 | 全局热度数据 |
| **新文章冷启动** | 基于内容的相似推荐 + 作者粉丝优先推荐 + 流量扶持池 | 文章元数据 |
| **新类别冷启动** | 类别关联推荐 + 交叉推荐 | 类别关联图谱 |

## 三、关键技术选型建议

### 3.1 推荐算法技术选型

| 技术方案 | 适用场景 | 优势 | 劣势 | 选型建议 |
|---------|---------|-----|-----|---------|
| **基于内容的推荐（Content-Based）** | 文章详情页关联推荐、新文章冷启动 | 实现简单、可解释性强、无冷启动问题 | 推荐多样性差、难以发现新兴趣 | ★★★ 基础方案，优先实现 |
| **协同过滤（Collaborative Filtering）** | 首页个性化推荐流 | 发现潜在兴趣、推荐多样性强 | 冷启动问题、数据稀疏问题 | ★★★ 核心方案 |
| **基于标签权重的混合推荐** | 搜索建议、分类页排序 | 平衡准确性与多样性、易于调参 | 标签质量依赖度高 | ★★★ 推荐首选混合方案 |
| **深度学习推荐（Deep Learning）** | 高精度排序、多目标优化 | 特征组合能力强、效果上限高 | 计算成本高、可解释性差 | ★★☆ 远期演进方向 |
| **图神经网络（GNN）** | 社交关系推荐、知识图谱推荐 | 关系建模能力强 | 实现复杂度高 | ★☆☆ 探索性方向 |
| **热度推荐（Hot Ranking）** | 新用户/未登录用户兜底 | 简单可靠、无需用户数据 | 无个性化 | ★★★ 兜底方案 |

### 3.2 推荐引擎技术栈

```
推荐引擎选型对比：

方案 A: 自研轻量引擎（推荐）
├── 召回: 基于 MySQL + Redis 的规则召回 + 简单协同过滤
├── 排序: 基于特征工程 + 加权公式
├── 优势: 与现有 Spring Boot 生态无缝集成、运维成本低
└── 劣势: 复杂模型能力有限

方案 B: Apache Mahout（中期可选）
├── 召回: 分布式协同过滤、聚类算法
├── 排序: 基于 Hadoop/Spark 的批量计算
├── 优势: 算法库丰富、可扩展
└── 劣势: 需要引入 Hadoop 生态、运维复杂度增加

方案 C: 云推荐服务（远期可选）
├── 例如: 阿里云推荐引擎、AWS Personalize
├── 优势: 开箱即用、免运维
└── 劣势: 成本高、数据隐私风险、定制化受限
```

**选型建议**: 初期采用**方案 A（自研轻量引擎）**，与现有 Spring Boot + MyBatis Plus + Redis + MySQL 技术栈深度整合。数据量增长后迁移至方案 B 或 C。

### 3.3 数据处理框架

| 环节 | 推荐技术 | 与现有技术栈的契合度 | 说明 |
|-----|---------|------------------|-----|
| **实时行为采集** | 前端自定义埋点 SDK + Spring Boot API | ★★★★★ | 基于现有 Vue 3 + Axios 架构扩展 |
| **消息队列** | Redis Stream / RabbitMQ | ★★★★☆ | 轻量级消息队列，降低架构复杂度 |
| **实时计算** | Spring 定时任务 + Redis | ★★★★★ | 初期使用定时任务替代 Flink，降低运维成本 |
| **离线计算** | Spring @Scheduled + MySQL 存储过程 | ★★★★★ | 离线特征计算和报表生成 |
| **搜索引擎** | Elasticsearch | ★★★☆☆ | 文章全文检索 + 标签检索（远期引入） |
| **日志收集** | Logstash + Elasticsearch + Kibana (ELK) | ★★★☆☆ | 行为日志分析（远期引入） |

### 3.4 存储方案细化

| 存储需求 | 推荐方案 | 理由 |
|---------|---------|-----|
| **结构化业务数据** | MySQL（现有） | 用户、文章、分类、标签等核心数据已使用 MySQL |
| **行为日志** | MySQL 分表 + 定期归档 | 避免引入新技术栈，分表保证写入性能 |
| **在线特征缓存** | Redis（现有） | 热门内容、用户实时特征、推荐结果缓存 |
| **用户画像** | MySQL JSON 字段 | 标签权重向量存储为用户表的 JSON 字段 |
| **全文搜索** | MySQL FULLTEXT / Elasticsearch（远期） | 先使用 MySQL 全文索引，规模增长后迁移 ES |
| **向量存储** | Redis + JSON 序列化（初期） | Embedding 向量大小可控时 Redis 够用 |

### 3.5 前端技术选型

| 需求 | 技术方案 | 说明 |
|-----|---------|-----|
| **推荐卡片组件** | Vue 3 Composition API + 现有组件库 | 基于现有 `ArticleInteraction.vue` 等组件扩展 |
| **无限滚动** | Intersection Observer API + 自定义 Hook | 替代第三方库，减少依赖 |
| **埋点 SDK** | 自研轻量 SDK | 基于 `navigator.sendBeacon` 保证数据不丢失 |
| **状态管理** | Pinia（现有 store） | 管理推荐列表状态、用户画像缓存 |
| **样式方案** | Tailwind CSS（现有） | 保持与项目一致的设计语言 |
| **A/B 测试分流** | 服务端 Cookie + localStorage | 前端读取实验分组标识，渲染对应组件 |

### 3.6 基础设施与扩展性

| 层次 | 当前状态 | 推荐方案 | 扩展策略 |
|-----|---------|---------|---------|
| **应用层** | Spring Boot 单体 | 保持单体，推荐模块作为独立 Service | 未来可拆分为微服务 |
| **缓存层** | Redis 单机 | 已有 Redis，增加推荐专用前缀 | 未来可 Redis Cluster |

## 六、潜在风险与应对策略

### 6.1 风险矩阵总览

```
影响程度
  ▲
  │  R4 隐私合规    │  R1 数据稀疏    │  R2 冷启动
  │  (中概率高影响)  │  (高概率高影响)  │  (高概率高影响)
  │                 │                 │
  ├─────────────────┼─────────────────┤
  │  R5 算法偏差    │  R3 性能瓶颈    │  R6 数据质量
  │  (低概率中影响)  │  (中概率中影响)  │  (中概率中影响)
  │                 │                 │
  └─────────────────┴─────────────────┴──► 发生概率
```

### 6.2 风险一：数据质量与稀疏性问题

| 维度 | 说明 |
|-----|-----|
| **风险描述** | WeBlog 初期用户量和内容量有限，用户-文章交互矩阵稀疏，导致协同过滤效果差 |
| **影响范围** | 个性化推荐准确率下降，推荐结果趋同于热门内容 |
| **发生概率** | **高**（初期 DAU < 1000 时几乎必然发生） |
| **影响程度** | **高**（直接影响推荐核心价值） |

**应对策略**:

| 策略 | 实施方式 | 优先级 |
|-----|---------|-------|
| **数据增强** | 利用用户注册时选择的兴趣标签作为冷启动特征；对低活跃用户使用类别偏好扩展 | P0 |
| **降级推荐** | 协同过滤结果不足时，自动降级为"内容推荐 + 热门推荐"混合策略 | P0 |
| **特征工程补充** | 引入文章元数据（标签、类别、作者、发布时间）作为内容推荐的强特征 | P1 |
| **跨场景数据复用** | 将搜索行为、收藏行为作为强信号，权重高于浏览行为 | P1 |
| **渐进式个性化** | 用户行为 < 10 条时使用热门推荐；10-50 条时使用内容推荐；> 50 条时启用协同过滤 | P1 |
| **数据积累计划** | 上线初期设置 2-4 周数据积累期，此阶段以内容推荐 + 热度推荐为主 | P2 |

---

### 6.3 风险二：冷启动问题

| 维度 | 说明 |
|-----|-----|
| **风险描述** | 新注册用户无历史行为数据，新发布文章无交互数据，导致推荐系统无法有效工作 |
| **影响范围** | 新用户体验差（千篇一律的热门推荐）；新文章得不到曝光（马太效应） |
| **发生概率** | **高**（任何 UGC 平台的固有挑战） |
| **影响程度** | **高**（直接影响新用户留存和新作者积极性） |

**应对策略**:

| 策略 | 实施方式 | 适用场景 |
|-----|---------|---------|
| **兴趣标签引导** | 新用户注册后引导选择 3-5 个感兴趣的标签，作为初始画像 | 新用户冷启动 |
| **探索-利用平衡（ε-greedy）** | 以 80% 概率推荐个性化内容，20% 概率随机推荐探索新内容 | 新用户冷启动 |
| **热门 + 新内容混合** | 新用户推荐流中 60% 热门 + 30% 新内容 + 10% 随机 | 新用户冷启动 |
| **基于内容的冷启动** | 新文章根据标签/类别匹配给感兴趣的已有用户 | 新文章冷启动 |
| **作者粉丝优先** | 新文章第一时间推送给作者的粉丝/历史读者 | 新文章冷启动 |
| **新内容流量池** | 新文章发布后自动进入"新内容流量池"，获得基础曝光量 | 新文章冷启动 |
| **编辑精选 + 人工推荐** | 优质新内容由运营手动推荐至显著位置 | 新文章冷启动 |

---

### 6.4 风险三：性能瓶颈

| 维度 | 说明 |
|-----|-----|
| **风险描述** | 用户量和内容量增长后，实时推荐计算压力增大，接口响应变慢，影响用户体验 |
| **影响范围** | 首页加载变慢、推荐接口超时、Redis 内存压力 |
| **发生概率** | **中**（用户量达到万级别时开始显现） |
| **影响程度** | **中**（可通过技术手段缓解，不影响核心功能） |

**应对策略**:

| 策略 | 实施方式 | 预期效果 |
|-----|---------|---------|
| **多级缓存架构** | L1: 本地 Caffeine 缓存（1min TTL）→ L2: Redis 缓存（1h TTL）→ L3: MySQL 查询 | P99 延迟降低 60%+ |
| **推荐结果预计算** | 离线批量计算每个用户的推荐列表，存入 Redis；在线请求直接返回缓存结果 | 接口响应 < 50ms |
| **异步非阻塞** | 行为日志写入异步化（MQ/Redis Stream），不影响主流程 | 写入 TPS 提升 3-5x |
| **数据库优化** | 行为日志表按月分表；推荐相关查询建立覆盖索引；读写分离 | 查询延迟降低 50% |
| **限流与熔断** | 推荐接口配置限流（令牌桶）；推荐服务不可用时降级为静态热门列表 | 避免雪崩 |
| **懒加载 + 分页** | 前端推荐流分页加载（每次 10 条），减少单次计算量 | 首屏加载 < 200ms |
| **计算资源扩展** | Redis 内存扩容；MySQL 只读副本；未来独立推荐计算节点 | 支撑 10x 流量增长 |

---

### 6.5 风险四：用户隐私合规

| 维度 | 说明 |
|-----|-----|
| **风险描述** | 收集用户行为数据用于个性化推荐，可能涉及《个人信息保护法》合规风险 |
| **影响范围** | 法律合规风险、用户信任度下降、可能的监管处罚 |
| **发生概率** | **中**（法规日益严格） |
| **影响程度** | **高**（可能导致功能下线或法律诉讼） |

**应对策略**:

| 策略 | 实施方式 | 优先级 |
|-----|---------|-------|
| **隐私政策透明化** | 明确告知用户收集哪些数据、用途是什么；提供清晰易懂的隐私政策 | P0 |
| **用户授权机制** | 首次使用推荐功能时弹窗获取用户同意（Opt-in）；提供"关闭个性化推荐"开关 | P0 |
| **数据最小化原则** | 仅收集推荐系统必需的数据字段；不采集敏感个人信息（如精确位置、设备唯一标识） | P0 |
| **数据脱敏与加密** | 行为日志中的用户标识使用加密 Token 替代明文 ID；传输层 HTTPS | P1 |
| **数据保留与删除** | 用户行为数据保留不超过 1 年；用户注销账户时关联行为数据同步删除 | P1 |
| **合规审查** | 上线前进行数据合规自评；必要时咨询法律顾问 | P1 |

---

### 6.6 风险五：算法偏差

| 维度 | 说明 |
|-----|-----|
| **风险描述** | 推荐算法可能产生"信息茧房"（过度个性化导致内容同质化）、"马太效应"（头部内容越来越热、新内容得不到曝光）、"偏见放大"（算法偏好某些类型内容） |
| **影响范围** | 用户体验下降、内容生态失衡、创作者公平性受损 |
| **发生概率** | **低**（初期内容量少时不易显现） |
| **影响程度** | **中**（长期影响平台内容生态健康） |

**应对策略**:

| 策略 | 实施方式 | 优先级 |
|-----|---------|-------|
| **多样性强制打散** | 推荐列表中相邻内容必须来自不同类别/作者；每页至少包含 1 条探索性内容 | P0 |
| **公平性监控** | 定期分析不同类别、不同发布时间的内容在推荐中的曝光分布；发现偏差及时调整权重 | P1 |
| **用户兴趣宽度监测** | 统计用户消费的内容类别数量变化；若持续收窄，增加探索性推荐比例 | P1 |
| **新内容保护机制** | 发布 24 小时内的文章获得 1.2x 推荐权重 | P1 |
| **人工审核抽样** | 定期人工抽查推荐结果，评估内容质量和多样性 | P2 |
| **用户反馈闭环** | "不感兴趣"反馈直接影响该类型内容的推荐权重 | P1 |

### C. 术语表

| 术语 | 说明 |
|-----|-----|
| **CTR (Click-Through Rate)** | 点击率 = 点击次数 / 曝光次数 |
| **CVR (Conversion Rate)** | 转化率 = 完成目标行为的次数 / 点击次数 |
| **协同过滤 (CF)** | 基于用户相似性或物品相似性进行推荐 |
| **冷启动 (Cold Start)** | 系统对新用户或新物品缺乏数据而难以推荐的问题 |
| **召回 (Recall)** | 从海量内容中快速筛选候选集的过程 |
| **排序 (Ranking)** | 对候选集进行精细排序的过程 |
| **EE (Explore-Exploit)** | 探索与利用平衡策略 |
| **P99 延迟** | 99% 的请求响应时间低于此值 |
